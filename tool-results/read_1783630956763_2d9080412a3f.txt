   805→   805→Capítulo 6 — Criterios de Aceptación
   806→   806→Los criterios de aceptación (CA-NNN) traducen cada requisito funcional a una condición verificable. Un criterio es
   807→   807→aceptable cuando puede responder de forma binaria (sí/no) a la pregunta "¿el sistema cumple este comportamiento
   808→   808→bajo estas condiciones?". Cada criterio se asocia a un requisito funcional y a un método de verificación: manual
   809→   809→(prueba exploratoria), E2E (Cypress/Playwright) o unit (Jest sobre Cloud Functions).
   810→   810→6.1 Criterios de aceptación por módulo
   811→   811→ID RF Criterio Verificación
   812→   812→CA-001 RF-00 Dado un invitado en /registro, cuando completa email válido y contraseña de 8+ E2E
   813→   813→1 caracteres y pulsa crear_cuenta, entonces se crea el usuario y se redirige a /onboarding.
   814→   814→CA-002 RF-00 Dado un email ya registrado, cuando el invitado intenta registrarse, entonces el sistema E2E
   815→   815→2 muestra "Ya existe una cuenta con este email".
   816→   816→CA-003 RF-00 Dado credenciales válidas, cuando el usuario pulsa iniciar_sesión, entonces se establece E2E
   817→   817→4 la sesión y se redirige a /foro.
   818→   818→CA-004 RF-00 Dado un email registrado, cuando el usuario solicita recuperar contraseña, entonces se Manual
   819→   819→6 envía un email con enlace de reseteo.
   820→   820→CA-005 RF-00 Dado 5 intentos fallidos consecutivos, cuando el usuario intenta un 6º login, entonces el Unit
   821→   821→7 sistema bloquea por 15 minutos.
   822→   822→CA-006 RF-00 Dado un usuario recién registrado, cuando accede a cualquier ruta protegida, entonces E2E
   823→   823→9 es redirigido a /onboarding.
   824→   824→CA-007 RF-01 Dado el paso 1 del onboarding, cuando el usuario ingresa nombre de 1 carácter, Unit

  1200→  1200→La estrategia de pruebas de AutoDev sigue el modelo de pirámide invertida: una base amplia de pruebas unitarias,
  1201→  1201→una capa intermedia de pruebas de integración y una capa superior reducida de pruebas end-to-end. Esta

  1203→  1203→La filosofía general es: si un bug puede detectarse con una prueba unitaria, no debe depender de una prueba E2E
  1204→  1204→para ser encontrado.

  1260→  1260→Capítulo 9 — Arquitectura del Sistema
  1261→  1261→9.1 Vista general
  1262→  1262→La arquitectura de AutoDev es 100% serverless sobre Firebase. No existen servidores aprovisionados ni
  1263→  1263→contenedores gestionados manualmente. El frontend (Next.js 16) se sirve desde Firebase Hosting con CDN global.
  1264→  1264→La autenticación delega a Firebase Auth. Los datos transaccionales viven en Firestore (NoSQL orientado a
  1265→  1265→documentos). La lógica de negocio sensible, los triggers reactivos y las tareas programadas se ejecutan en Cloud
  1266→  1266→Functions v2. Los archivos binarios (avatares, imágenes de recursos) se alojan en Cloud Storage for Firebase. Las
  1267→  1267→notificaciones push se entregan vía FCM. La analítica de producto se captura con Firebase Analytics y se visualiza
  1268→  1268→en Google Analytics 4.
  1269→  1269→Esta elección arquitectónica permite escalar a 10.000 usuarios concurrentes sin re-arquitectura, elimina el costo de
  1270→  1270→servidores ociosos (pay-per-use) y descarga al equipo de tareas operativas como parcheo de SO o gestión de
  1271→  1271→certificados TLS. El compromiso es el lock-in a Google Cloud, mitigado por el patrón Repository en el frontend
  1272→  1272→que aísla las llamadas a Firebase detrás de una interfaz intercambiable.

  1297→  1297→9.3 Patrones arquitectónicos
  1298→  1298→9.3.1 Repository Pattern en el frontend
  1299→  1299→Todas las llamadas a Firebase desde el frontend se encapsulan detrás de interfaces Repository. Por ejemplo,
  1300→  1300→PostRepository expone métodos como create(), list(), comment() y like() que internamente usan el SDK de
  1301→  1301→Firestore. Este patrón facilita mockear Firebase en pruebas unitarias del frontend y permite, en el futuro, migrar a
  1302→  1302→otro backend sin reescribir componentes. Los repositorios se inyectan vía React Context.
  1303→  1303→9.3.2 Event-driven backend con triggers de Firestore
  1304→  1304→La lógica reactiva se implementa mediante Cloud Functions reactivas a eventos de Firestore. Cuando un documento
  1305→  1305→se crea o actualiza, Firestore emite un evento que dispara la función. Por ejemplo, onPostCreate se dispara al crear
  1306→  1306→un documento en /posts, valida el contenido, otorga XP al autor y actualiza contadores. Este patrón desacopla la
  1307→  1307→escritura de datos (frontend directo a Firestore) del enriquecimiento y validación de negocio (Cloud Functions).
  1308→  1308→9.3.3 Serverless con escalado a cero
  1309→  1309→Cloud Functions v2 escala automáticamente de 0 a N instancias según la carga. El cold start es el principal desafío;
  1310→  1310→se mitiga configurando minInstances=1 en funciones críticas (auth, onboarding) y usando funciones más ligeras (sin
  1311→  1311→dependencias pesadas) para triggers frecuentes. Las funciones HTTP se exponen detrás de Firebase Hosting para
  1312→  1312→aprovechar su CDN y reducir latencia.
  1313→  1313→9.3.4 Single source of truth en Firestore Rules
  1314→  1314→Las reglas de seguridad de Firestore son la única fuente de verdad para autorización. El frontend puede ocultar
  1315→  1315→botones o rutas según el rol del usuario, pero esa lógica es cosmética; el backend (Firestore Rules) rechaza cualquier
  1316→  1316→acceso no autorizado. Esta separación evita vulnerabilidades donde un usuario malicioso podría eludir la UI para
  1317→  1317→acceder a datos.
  1318→  1318→9.4 Flujos de datos principales
  1319→  1319→9.4.1 Flujo: Publicación de un post
  1320→  1320→El Miembro completa el formulario en el frontend. El frontend valida y sanitiza localmente, luego escribe
  1321→  1321→directamente en /posts con set() generando un ID automáticamente. La escritura pasa por las reglas de Firestore que
  1322→  1322→validan: (i) el usuario está autenticado, (ii) tiene status="active", (iii) los campos obligatorios están presentes y
  1323→  1323→cumplen longitud máxima. Si las reglas pasan, el documento se persiste. Inmediatamente, el trigger onPostCreate se
  1324→  1324→dispara en Cloud Functions: la función valida nuevamente (defensa en profundidad), otorga 25 XP al autor (update
  1325→  1325→en /users/{uid}), incrementa postsCount del autor y publica un evento en Analytics.
  1326→  1326→9.4.2 Flujo: Cálculo del ranking semanal
  1327→  1327→Una Cloud Function scheduled se ejecuta cada lunes a las 00:00 UTC-4. La función consulta /users ordenado por
  1328→  1328→weeklyXP descendentemente, limit 100. Escribe el resultado en /rankings/{year-Wxx} como snapshot inmutable.
  1329→  1329→Otorga XP bonus a los top 3 (500, 300, 200). Resetea el campo weeklyXP de todos los usuarios a 0. Genera
  1330→  1330→notificaciones in-app para los top 100 anunciando su posición.

  1356→  1356→Capítulo 10 — Diseño de Base de Datos
  1357→  1357→(Firestore)
  1358→  1358→Firestore es una base de datos NoSQL orientada a documentos organizada en colecciones y subcolecciones. A
  1359→  1359→diferencia de SQL, no hay joins ni esquema rígido; la desnormalización controlada es una práctica recomendada
  1360→  1360→para optimizar lecturas. Este capítulo documenta cada colección, su esquema, índices compuestos necesarios, reglas

  1766→  1766→Capítulo 11 — Roadmap y Entregables
  1767→  1767→El roadmap de AutoDev se organiza en cuatro fases incrementales, cada una con criterios de salida verificables. La
  1768→  1768→filosofía es entregar valor temprano: el MVP (fase 1) ofrece una comunidad funcional mínima que permite validar
  1769→  1769→la propuesta de valor con usuarios reales antes de invertir en funcionalidades avanzadas. Cada fase se planifica con
  1770→  1770→un horizonte de 6 a 10 semanas de desarrollo por un equipo de 3-5 ingenieros.
  1771→  1771→11.1 Fases del roadmap
  1772→  1772→Fase Duración Módulos incluidos Criterio de salida
  1773→  1773→Fase 1 — MVP 8 semanas Auth, Onboarding, Foro, Miembros 100 usuarios beta activos, NPS > 30, 0
  1774→  1774→bugs críticos
  1775→  1775→Fase 2 — Contenido 6 semanas Recursos, Cursos, Gamificación (misiones, 50 recursos publicados, 5 cursos activos,
  1776→  1776→XP) retención D7 > 40%
  1777→  1777→Fase 3 — Engagement 6 semanas Directos, Ranking, Notificaciones push Leaderboard operativo, 1000 push
  1778→  1778→enviadas, retención D30 > 25%
  1779→  1779→Fase 4 — Escala 8 semanas Panel Admin, Analytics avanzado, Stripe Admin operativo, dashboard métricas,
  1780→  1780→10 pagos procesados
  1781→  1781→11.2 Criterios de salida por fase
  1782→  1782→Cada fase tiene criterios de salida objetivos y medibles. Ninguna fase puede considerarse completada hasta que
  1783→  1783→todos sus criterios estén verificados. Esto evita la deuda técnica acumulada por "lo arreglamos en la siguiente fase".
  1784→  1784→l Fase 1 MVP: 100 usuarios beta registrados con onboarding completo; al menos 1 post y 1 comentario por
  1785→  1785→usuario en promedio; NPS superior a 30; 0 bugs críticos abiertos; Lighthouse Performance > 80; cobertura
  1786→  1786→de pruebas > 70%.
  1787→  1787→l Fase 2 Contenido: 50 recursos publicados por al menos 5 autores distintos; 5 cursos activos con 20
  1788→  1788→lecciones totales; retención D7 superior al 40%; sistema de XP operativo y verificado por pruebas de carga.
  1789→  1789→l Fase 3 Engagement: Leaderboard semanal operativo con 4 semanas de histórico; 1000 notificaciones
  1790→  1790→push entregadas con tasa de apertura > 20%; retención D30 superior al 25%; al menos 3 directos realizados
  1791→  1791→con 50 asistentes promedio.
  1792→  1792→l Fase 4 Escala: Panel de admin usado semanalmente por al menos 2 administradores; dashboard de
  1793→  1793→métricas con 10 KPIs en tiempo real; 10 pagos procesados con Stripe sin discrepancias; documentación
  1794→  1794→runbooks completa para incidentes.
